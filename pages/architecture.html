Data Flow

1

Ingestion

Metrics arrive via Prometheus remote_write, logs via HTTP POST, and deployments via GitHub webhooks. All data is timestamped and queued in Redis streams for processing.

2

Detection

Background workers consume the streams. The anomaly detector compares current values against rolling baselines using z-score analysis. Anomalies exceeding the threshold (default 2.5Ïƒ) are flagged.

3

Analysis

Flagged anomalies are enriched with context (recent deployments, related logs, historical incidents) and sent to the local LLM for root cause analysis. The AI returns a diagnosis and suggested actions.

4

Action Proposal

Suggested actions are evaluated against safety rules (blast radius, rate limits). Valid actions are queued for approval or auto-executed based on the operating mode (supervised vs autonomous).

5

Execution

Approved actions are executed via the appropriate provider (kubectl, cloud SDK, database client). Pre and post-execution metrics are captured for impact assessment.

6

Learning

Outcomes (success, failure, rollback) are fed back into the learning engine. Successful patterns increase confidence scores; failures trigger alerts and lower future auto-approval thresholds.

Key Components

API Server (FastAPI)

RESTful API for dashboard, integrations, and data ingestion. Includes authentication, rate limiting, and request validation.

Background Workers

Async processors for anomaly detection, AI analysis, and action execution. Scaled independently based on queue depth.

Ollama (LLM)

Local AI inference for privacy. Supports Llama 3, Mistral, and custom fine-tuned models. No external API calls.

Action Executor

Sandboxed execution environment with timeout handling, rollback capabilities, and comprehensive audit logging.
